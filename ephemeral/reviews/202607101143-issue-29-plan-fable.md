## Findings

### 1. design: proof commands omit the repository's own generation gate, which is the primary activation surface

The plan's proof section stops at `go test ./...`. But `go test ./...` never
regenerates anything — it compiles and tests **checked-in** generated code
(`AssertGoldenFile` only diffs builder fixture output in `test_run/`). The
real-world activation surface for this change is the examples tree:

- `examples/structs/types.go:51` — `BirthDate time.Time` on a registered type;
- `examples/uniontypes/types.go:76,169` — two more exported `time.Time`
  fields.

Without the `time.Time` traversal leaf, `go generate ./...` on those examples
is exactly where the fix detonates (stdlib `time` gets `packages.Load`-ed and
scanned); with the leaf, it is the strongest end-to-end proof the plan can run
— and the repo *already defines this gate in CI*
(`.github/workflows/go.yml:25-31`: `go generate ./...` → clean-tree check →
`JSONSCHEMA_NO_CHANGES=1 go generate ./...` → `go test ./...`), and
`docs/design/issue-28-fable-review.md` §9.7 names it explicitly. The plan's
proof commands should include the same sequence locally:

```sh
go generate ./...
test -z "$(git status --porcelain)"
JSONSCHEMA_NO_CHANGES=1 go generate ./...
go test ./...
```

This also forces inspection of any example-schema diffs the traversal change
legitimately produces, instead of discovering them as a dirty-tree CI failure
after merge.

### 2. design: newly reachable zero-TypeSpec panic for remote registered enum/interface field types — untested and unmentioned

`resolveTypes` has two remote paths that classify differently:

- **Fresh-load path** (`scan_result.go:637-645` →
  `loadPackageInternal:477-490`): requested remote names are checked against
  `LocalNamedTypes`, then `Constants`, then `Interfaces` — enums and
  interfaces are handled.
- **Already-in-deps path** (`scan_result.go:628-635`): blindly does
  `remote.resolveQueue = append(..., remote.LocalNamedTypes[typeName])`.
  Registered enums/interfaces are *not* in `LocalNamedTypes`
  (`scan_result.go:437-444`), so this appends a zero `TypeSpec`, and the queue
  loop dereferences `ts.Concrete.Name.Name` (`scan_result.go:619`) — nil
  pointer panic. Note also `alreadyTraversedLocally` is never set for
  enum/interface names on the fresh path, so the guard at line 630 never
  saves you on a second reference.

Pre-fix this is nearly unreachable (exported fields are never traversed, and
top-level named remote-enum types as in the `enums` fixture always arrive via
one batched fresh load). Post-fix, an exported struct field typed as a remote
registered enum (e.g. `Status enumsremote.RemoteEnumType`) hits the deps path
whenever the remote package is already loaded — which depends on `remoteTypes`
**map iteration order** (line 627), i.e. a *flaky* panic. The plan's fixture
(step 6) covers only a remote **struct** field plus `time.Time`; its test
matrix has no remote-enum or twice-referenced-remote-package case, and the
plan text never mentions the deps-path asymmetry. Either mirror the
`Constants`/`Interfaces` exemption in the deps branch as part of this change,
or add the remote-enum-field case to the fixture and record the boundary
explicitly if it is deferred.

### 3. design: registered *remote* interfaces as field types will hard-error post-fix; the plan should handle or explicitly scope this out

The legacy API registers remote interfaces from the consuming package:
`typescanner/calls.go:42` — `NewInterfaceImpl[scannersubpkg.MarkerInterface](...)`
— and `scannersubpkg` itself has **no** registration file. The plan's
classification step 4 consults `r.Interfaces[expr.Name]` only in the
*local-ident* branch; a field typed `scannersubpkg.MarkerInterface` arrives as
an ident with a non-local `Path`, bypasses that check, is added to
`remoteTypes`, and the remote fresh load then fails with `undeclared local
type found: MarkerInterface` (`scan_result.go:485`) because the interface is
neither a named type, constant, nor registered interface *in the remote
package*. Pre-fix such a field is silently skipped; post-fix it becomes a
generation failure. Issue 29's text and the plan's goal say "registered
**local** interfaces", so deferring is defensible — but since the fixture
registry proves remote-interface registration is a supported shape, the plan
should either extend the leaf check to path-qualified registered interfaces
(note: `r.Interfaces` is keyed by bare `TypeName` at `scan_result.go:396`,
losing the package path — the check would need the `TypeID`) or add one
sentence to Non-goals plus a tracking note, so the failure mode is a decision
rather than a surprise.

### 4. nit: the `time.Time` leaf predicate has a second remote-registration site to cover

The plan says "before adding an external identifier to `remoteTypes`" —
singular. There are two registration sites: the `*dst.Ident` branch
(`scan_result.go:545`) and the `*dst.SelectorExpr` fallback
(`scan_result.go:547-558`, including the line-554 path that uses the package
*name* as the map key). With the resolver active, `time.Time` should always
arrive as a path-qualified ident, but the fallback exists precisely for the
cases where that assumption fails. Apply the predicate in both branches (or
assert the invariant) so the leaf boundary can't be bypassed; note the
renderer's special case is likewise ident-only, so the two stay symmetric.

### 5. nit: prefer `go/token.IsExported`; the consolidation leaves a third copy of the visibility rule in `PropNames`

The plan already says "not the absence of a lowercase rune" — good. Prefer
`token.IsExported` (decodes the first *rune*) over the repo's recurring
`unicode.IsUpper(rune(name[0]))` byte-index pattern, which mis-handles
multi-byte identifiers and appears in `skipField` (`scan_result.go:588`),
`StructField.Skip` (`node_wrappers.go:693`), **and** `StructField.PropNames`
(`node_wrappers.go:659`). Decision 1 routes only the first two through the
shared helper; fold `PropNames` in too (or note why not), otherwise the
"parallel implementations" smell the plan cites from review §6.4 survives in a
third location.

### 6. nit: proof-command test names are speculative

`TestSkipField|TestRegisteredInterfaceIdentifierResolves|TestUnknownLocalIdentifierFails`
match tests that don't exist yet; the plan hedges for the builder pattern but
not the syntax one. Harmless as long as the final names are back-filled into
the plan/PR description — the issue's own
`TestSkipFieldExportVisibilityContract` is already matched by the
`TestSkipField` prefix.

## Verdict

The plan is accurate against the repository at `13421b6`: every cited line,
behavior, and fixture claim checked out; the two-inversions-one-unit scoping
is correct and independently confirmed by `FancyStruct.IFace`; the skip-policy
split matches the real `skipField`/`Skip()` divergence; the non-goals keep
issue 28/30 scope out cleanly; nothing is over-engineered. The gaps are in
proof and blast-radius edges, not direction: add the repo's generation gate to
the proof commands (finding 1), and decide the two remote-registered-type
edges (findings 2-3) before coding rather than during. With those addressed,
implementation is safe to begin.
