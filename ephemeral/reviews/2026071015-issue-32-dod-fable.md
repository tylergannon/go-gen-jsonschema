### design

1. **The OpenAI proof drops the round-trip half of the issue's claim.**
   Issue #32 proof-of-work requires: "A schema containing only ordinary and
   supported Nullable fields is accepted by OpenAI strict schema validation
   **and returned JSON passes generated validation**." The DoD's Behavioral
   proof section requires only submitting the schema and attaching redacted
   request/response evidence. Feeding the model's returned JSON through the
   generated `ValidateJSON` (and the wrapper unmarshal path) is the one step
   that demonstrates the library's headline use case — LLM structured output
   consumed by generated Go code — actually works with the new Nullable
   encodings. As written, the live test proves schema *acceptance* only. Fix
   is one sentence: the recorded evidence must show the API's returned JSON
   passing the example's generated validation (and ideally decoding into the
   wrapper types).

2. **Leftover legacy-tag semantics are not pinned down.** Issue decision 5
   explicitly decides: "A leftover legacy tag has no optionality effect" —
   i.e., a stray `jsonschema:"optional"` becomes inert, not a generation
   error. The DoD says only "The legacy `jsonschema:"optional"` behavior and
   its documentation are removed as the agreed breaking change." Two
   contradictory implementations satisfy that sentence: silently ignoring the
   tag (the decided contract) or rejecting it with a generator error (not
   decided, and a different breaking change for downstream consumers). Since
   the DoD positions itself as the standalone completion contract, it should
   state the decided leftover-tag behavior, and — because the tag is still
   live in `examples/test_options` and two builder fixtures — that those
   usages stop asserting optionality effects. Ambiguous completion criterion.

### nit

1. The issue's completion contract requires the PR to include "a tracked
   session worklog"; the DoD's Hygiene and delivery section omits it.
2. The issue requires "The exact pull-request head has green CI." The DoD's
   Hygiene section lists only local `go test ./...`; exact-head CI appears
   only obliquely as "exact-head proof" inside review checkpoint 2. Make it an
   explicit hygiene bullet.
3. Issue decision 14 (all integer variants including `byte`/`rune`/`uintptr`
   render as integers for ordinary **and wrapped** fields; floats as numbers
   with normal NaN/Inf rejection) has no corresponding acceptance line or test
   mention in the DoD. The blanket "scalar and named scalar values" plausibly
   covers it, but one sentence under unit-test lockdown would remove the doubt.
4. Issue decision 16 enumerates what README/`llms.txt`/skill must agree on
   (three contracts, selection guidance, `omitzero`, the generated-validation
   missing-vs-null caveat "said plainly," supported/rejected shapes, interface
   behavior, strict-schema implications, usable examples). The DoD compresses
   this to "describe the behavior that actually shipped," and states the
   missing-vs-null caveat as a fact without carrying it into the documentation
   requirement. Decision 13's "the manual schema-building API remains
   unchanged" is likewise not carried over. Low risk, but the DoD is the
   standalone contract and these are cheap to restate.

## Conclusion

The direction is concrete, faithful to the decided feature, and appropriately
scoped; the proof design (real CLI generation with validation, compiled proof
command that exits non-zero, committed deterministic transcript, failures
exercised through the real generator, live OpenAI evidence rather than a local
stand-in) would genuinely demonstrate the public generator and generated
consumer behavior. The two design findings — restoring the returned-JSON →
generated-validation leg of the OpenAI proof, and pinning the decided inert
behavior of leftover legacy tags — are each one-sentence amendments to the DoD
and should be fixed before implementation proceeds against it. No simpler
equally-conclusive proof was found; the live API step is required by the issue
text and cannot be replaced by a local validator without weakening the claim.
